Pain Point02/14

Quy trình đặt cơm thủ công tạo nhiều chi phí ẩn

Vấn đề không chỉ là chọn món; vấn đề chính là điều phối, tổng hợp và kiểm soát trạng thái.

Người phụ trách

Phải nhắc nhiều lần, đọc tin nhắn rời rạc, tổng hợp đơn, chia tiền và kiểm tra thanh toán thủ công.

Thành viên nhóm

Dễ quên chọn món, ghi thiếu note, chọn món không còn bán, hoặc không biết tổng tiền cần trả.

Rủi ro vận hành

Sai món, thiếu đơn, sai tổng bill, trễ deadline đặt hàng, mất thời gian follow-up.

Insight: Đây là bài toán workflow state management, không phải chỉ là chatbot hỏi đáp.
Problem Statement03/14

Làm sao tự động hóa luồng đặt cơm mà vẫn kiểm soát được dữ liệu thật?

Hệ thống cần hiểu yêu cầu tự nhiên, gọi đúng tool, cập nhật trạng thái và xuất kết quả có thể kiểm chứng.

Problem statement: Trong nhóm nhỏ, quy trình đặt cơm trưa hằng ngày bị phân mảnh qua chat. Cần một assistant có khả năng kiểm tra menu thật, ghi nhận đơn, phát hiện người chưa chọn, tổng hợp bill và theo dõi thanh toán để giảm lỗi điều phối.

Solution Direction04/14

Baseline chatbot không đủ; ReAct agent phù hợp hơn

Prototype cố tình so sánh hai hướng để chứng minh giới hạn của LLM-only chatbot.

Baseline Chatbot

  • Trả lời trực tiếp từ LLM.
  • Có conversation history.
  • Không gọi tool, không cập nhật order state.
  • Dễ hallucinate món, giá, trạng thái.

ReAct Agent

  • Suy nghĩ theo vòng Thought → Action → Observation.
  • Gọi menu/order/bill/user tools.
  • Không invent data; dùng kết quả tool.
  • Có logging, metrics và error handling.

Code evidence: src/chatbot/chatbot.py không có tool calling; src/agent/agent.py triển khai parser, executor và max-step loop.

Demo Script13/14

Kịch bản thuyết trình prototype

Dùng 3 prompt để chứng minh sự khác biệt giữa chatbot và agent.

Prompt 1

“Gợi ý món dưới 50k không cay” → Agent gọi search_menu và trả món từ menu thật.

Prompt 2

“Thêm một đơn cho Minh: Cơm gà, ít cơm” → Agent gọi add_order và lưu trạng thái.

Prompt 3

“Ai chưa chọn món?” → Agent dùng check_missing_orders để đối chiếu members với orders.

Message chính: Chatbot trả lời; agent thực hiện công việc.
Conclusion14/14

Kết luận

Prototype đã chứng minh đúng hướng: lunch ordering là bài toán agentic workflow có tool interaction.

1. Pain point thật nằm ở điều phối và kiểm soát trạng thái, không chỉ ở giao tiếp.

2. ReAct Agent phù hợp hơn baseline chatbot vì có thể gọi tool, đọc observation và trả kết quả grounded.

3. Prototype đã có core workflow, tool layer, telemetry và 40 offline tests passed.

4. Version 1 cần tập trung vào persistence, structured output, UI/channel integration, security và monitoring.
